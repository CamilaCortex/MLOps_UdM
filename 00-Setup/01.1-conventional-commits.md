## Conventional Commits — Cheat Sheet

Referencia completa: <https://www.conventionalcommits.org/en/v1.0.0/>

### ¿Por qué importa?

Cuando trabajas en equipo (o incluso solo), los mensajes de commit son el historial de tu
proyecto. Si todos escriben lo que quieren, el historial se vuelve ilegible. Conventional
Commits es una convención simple que resuelve eso: cada mensaje sigue un formato predecible.

### Formato

```
<tipo>(<alcance opcional>): <descripción corta>
```

- Todo en **minúsculas**.
- La descripción empieza en minúscula y **no** lleva punto final.
- Máximo ~72 caracteres en la primera línea.

### Tipos principales

| Tipo | Cuándo usarlo | Ejemplo |
|------|---------------|---------|
| `feat` | Agregas funcionalidad nueva | `feat: add data validation pipeline` |
| `fix` | Corriges un bug | `fix: resolve null pointer in preprocessing` |
| `docs` | Solo cambias documentación | `docs: update setup instructions` |
| `refactor` | Cambias código sin agregar funcionalidad ni corregir bugs | `refactor: simplify feature engineering logic` |
| `test` | Agregas o corriges tests | `test: add unit tests for model training` |
| `chore` | Tareas de mantenimiento (dependencias, CI, configs) | `chore: update pip dependencies` |
| `ci` | Cambios en CI/CD (GitHub Actions, pipelines) | `ci: add linting step to workflow` |
| `style` | Formato, espacios, punto y coma (sin cambio de lógica) | `style: fix indentation in train.py` |

### Alcance (opcional)

El alcance indica **qué parte** del proyecto se ve afectada. Va entre paréntesis:

```
feat(api): add endpoint for predictions
fix(data): handle missing values in dataset
docs(setup): add GPG key instructions
```

No hay una lista fija de alcances — úsalo cuando aporte claridad.

### Ejemplos rápidos de commits

```bash
# Nueva funcionalidad
git commit -m "feat: add model evaluation metrics"

# Bug fix
git commit -m "fix: correct label encoding for categorical features"

# Documentación
git commit -m "docs: add conventional commits cheat sheet"

# Refactor
git commit -m "refactor(training): extract hyperparameter config to yaml"

# Tests
git commit -m "test: add integration tests for data pipeline"

# Mantenimiento
git commit -m "chore: upgrade scikit-learn to 1.4"

# CI/CD
git commit -m "ci: add model validation step to GitHub Actions"
```

### Para títulos de Pull Requests

Usa el mismo formato. El título del PR describe **qué hace el conjunto de cambios**:

```
feat(monitoring): add data drift detection with Evidently
fix(pipeline): resolve timeout in batch prediction step
docs(setup): add SSH and GPG configuration guide
```

### Para descripción de Pull Requests

El cuerpo del PR es libre, pero una buena estructura mínima es:

```markdown
## Qué cambia
- Breve lista de lo que se hizo.

## Por qué
- Una línea explicando la motivación.

## Cómo probarlo
- Pasos para verificar que funciona.
```

### Nombres de ramas

Las ramas siguen una lógica parecida a los commits: un **tipo** seguido de una descripción
corta separada por `/`. Usa guiones (`-`) para separar palabras.

```
<tipo>/<descripción-corta>
```

| Tipo | Cuándo usarlo | Ejemplo |
|------|---------------|---------|
| `feat/` | Nueva funcionalidad | `feat/add-data-validation` |
| `fix/` | Corrección de un bug | `fix/null-pointer-preprocessing` |
| `docs/` | Cambios en documentación | `docs/setup-guides` |
| `refactor/` | Reestructuración de código | `refactor/training-pipeline` |
| `chore/` | Mantenimiento, dependencias | `chore/upgrade-dependencies` |
| `ci/` | Cambios en CI/CD | `ci/add-linting-workflow` |
| `test/` | Agregar o corregir tests | `test/model-evaluation` |

Reglas:

- Todo en **minúsculas**, sin espacios.
- Corto y descriptivo: que se entienda de qué trata sin abrir el código.
- Evita nombres genéricos como `fix/bug` o `feat/update`.

Ejemplos completos del flujo:

```bash
# Rama
git checkout -b feat/add-drift-detection

# Commits dentro de esa rama
git commit -m "feat(monitoring): add drift detection with Evidently"
git commit -m "test(monitoring): add tests for drift detection"

# Título del PR
# feat(monitoring): add data drift detection with Evidently
```

### Errores comunes

| Mal | Bien | Por qué |
|-----|------|---------|
| `Updated stuff` | `fix: correct data path in config` | Sé específico |
| `feat: Add new feature` | `feat: add new feature` | Minúscula después de `:` |
| `fix: fixed the bug.` | `fix: resolve null check in loader` | Sin punto final, sin "fixed" |
| `changes` | `refactor: simplify model selection logic` | Usa un tipo y describe qué |
| `WIP` | No hagas commit de trabajo incompleto | Usa `git stash` si necesitas guardar progreso temporal |

### Regla de oro

Antes de escribir el mensaje, pregúntate:

> Si alguien lee solo este mensaje dentro de 6 meses, ¿entendería qué se hizo y por qué?
